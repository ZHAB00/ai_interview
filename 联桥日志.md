## 发现的问题与解决方案

### 问题 1：Refresh Token 请求格式不匹配 🔴 高

**发现**: 前端 `authStore.js` 和 `authService.js` 在调用 `POST /api/auth/refresh` 时，将 refresh_token 放在 `Authorization` 请求头中（`Bearer <refresh_token>`），请求体为空 `{}`。然而后端 `auth.py` 期望从请求体中读取 `refresh_token` 字段（`RefreshRequest.refresh_token`）。

**影响**: Token 刷新功能完全失效，用户登录 24 小时后必须重新登录。

**修复**: 修改前端两个文件，改为在请求体中发送 `{ refresh_token: refreshToken }`。

- `vue_ai_interview/src/stores/authStore.js` — 第42行
- `vue_ai_interview/src/services/authService.js` — 第12行

```diff
- api.post('/api/auth/refresh', {}, {
-   headers: { Authorization: `Bearer ${refreshToken}` }
- })
+ api.post('/api/auth/refresh', { refresh_token: refreshToken })
```

### 问题 2：反馈接口 URL 不匹配 🔴 高

**发现**: 前端 `interviewService.js` 中 `submitFeedback` 函数调用 `POST /api/reports/{reportId}/feedback`，但后端 `feedback.py` 的路由为 `POST /api/feedback/{interview_id}`。路径前缀和参数名均不匹配。

**影响**: 用户在报告页提交评分纠正反馈时会收到 404 错误。

**修复**: 修改前端 `interviewService.js`，将 URL 从 `/api/reports/{reportId}/feedback` 改为 `/api/feedback/{interviewId}`，同时将参数名从 `reportId` 改为 `interviewId` 以与后端一致。

```diff
- export function submitFeedback(reportId, feedback) {
-   return api.post(`/api/reports/${reportId}/feedback`, feedback)
- }
+ export function submitFeedback(interviewId, feedback) {
+   return api.post(`/api/feedback/${interviewId}`, feedback)
+ }
```

### 问题 3：WebSocket URL 相对路径问题 🔴 高

**发现**: 后端 `POST /api/interviews` 返回的 `ws_url` 为相对路径 `/ws/interview/{id}`（如 `/ws/interview/123`）。前端 `InterviewRoom.vue` 从 sessionStorage 读取该值后，直接传给 `useWebSocket.connect(wsUrl, token)`，后者执行 `new WebSocket(\`${wsUrl}?token=${token}\`)`。

在开发环境中，前端运行在 `localhost:5173`（Vite），后端运行在 `localhost:8000`（FastAPI）。浏览器会将相对路径 WebSocket URL 解析为 `ws://localhost:5173/ws/interview/123`，而 WebSocket 服务实际运行在 `ws://localhost:8000/ws/interview/123`，导致连接失败。

**影响**: 面试核心功能（实时对话）完全不可用。

**修复**: 修改前端 `InterviewRoom.vue`，从 `VITE_API_BASE_URL` 环境变量推导 WebSocket 基础 URL（`http` → `ws`），拼接后端返回的相对路径得到完整 URL。

```diff
+ const apiBase = import.meta.env.VITE_API_BASE_URL || ''
+ const wsBase = apiBase.replace(/^http/, 'ws')
+ const wsUrl = wsBase + wsPath
```

### 问题 5：缺少 `POST /api/auth/logout` 登出接口 🟢 低

**发现**: 前端 `authService.js` 定义了 `logout()` 函数调用 `POST /api/auth/logout`，但后端 `auth.py` 未实现此路由。注意：前端的 `authStore.logout()` 实际上并未调用 `authService.logout()`，仅在本地清除 token，因此该问题当前不阻塞功能。但为完整性和未来扩展考虑，应当补齐。

**影响**: 当前无实际影响（前端 store 的 logout 不调用 API）。但 `authService.logout()` 被定义却无对应后端路由，属于接口契约缺口。

**修复**: 在后端 `auth.py` 新增 `POST /api/auth/logout` 路由，返回 204 No Content。JWT 本身无状态，此端点作为预留接口供未来服务端 token 黑名单等扩展使用。

```python
@router.post("/logout", status_code=204)
async def logout():
    """用户登出。JWT 无状态，登出由客户端清除 token 完成。"""
    return None
```

### 问题 6：useReport.js 函数名遮蔽导致递归调用 🟡 中

**发现**: `useReport.js` 中本地定义的 `async function submitFeedback(errorIndex, feedback)` 函数名与从 `interviewService.js` 导入的 `submitFeedback` 同名，导致本地函数体内的 `await submitFeedback(...)` 实际调用自身而非导入的服务函数，造成无限递归。

**影响**: 用户在报告页提交反馈时会栈溢出，功能不可用。

**修复**: 将导入重命名为 `submitFeedbackApi`，本地 wrapper 函数调用 `submitFeedbackApi`：

```diff
- import { getInterviewReport, submitFeedback } from '../services/interviewService.js'
+ import { getInterviewReport, submitFeedback as submitFeedbackApi } from '../services/interviewService.js'

  async function submitFeedback(errorIndex, feedback) {
-   await submitFeedback(interviewId, { error_index: errorIndex, ...feedback })
+   await submitFeedbackApi(interviewId, { error_index: errorIndex, ...feedback })
  }
```

### 问题 8：Quick Start 快速体验 — resume_id 类型不匹配 + 缺失简历 🔴 高

**发现**: 前端 DashboardPage "使用示例简历快速体验" 流程存在两个连锁问题：

1. **类型错误**: `resumeId.value = 'demo_resume'` 将 resume_id 设为字符串，但后端 `CreateInterviewRequest.resume_id` 定义为 `int`。Pydantic 校验直接拒绝此请求，返回 422 错误。
2. **数据缺失**: 即便改为整数，数据库中也不存在该虚构简历，后端会返回 404 "简历不存在"。

快速体验的本意是让用户跳过简历上传直接体验面试，但后端设计强制要求一个真实的简历记录。

**影响**: 快速体验功能完全不可用，创建面试按钮点击后提示"创建面试失败"。

**修复**: 前后端协同修改：

- **后端 `schemas/interview.py`**: `resume_id` 从 `int` (必填) 改为 `int | None = None` (可选)
- **后端 `api/v1/interviews.py`**: 当 `resume_id` 为 None 时，自动创建占位简历（含 `quick_start: True` 标记），使用 `await db.flush()` 获取临时 ID
- **前端 `DashboardPage.vue`**: Quick Start 模式将 `resumeId.value` 设为空字符串，`handleStartInterview` 仅在 `resumeId` 有效且非 demo 时才发送 `resume_id` 字段

```diff
# 后端 schema
- resume_id: int
+ resume_id: int | None = None

# 后端 interview creation
+ if req.resume_id is not None:
+     # verify resume ownership...
+     resume_id = resume.id
+ else:
+     placeholder = Resume(user_id=..., file_path="", parsed_data={...}, ...)
+     db.add(placeholder)
+     await db.flush()
+     resume_id = placeholder.id

# 前端 Quick Start
- resumeId.value = 'demo_resume'
+ resumeId.value = ''

# 前端 create params
- const params = { resume_id: resumeId.value, position, difficulty, mode }
+ const params = { position, difficulty, mode }
+ if (resumeId.value && resumeId.value !== 'demo_resume') {
+   params.resume_id = resumeId.value
+ }
```

### 问题 9：422 校验错误响应格式不统一 🟡 中

**发现**: Pydantic `RequestValidationError` (422) 的默认响应格式为 `{ "detail": [{"loc": [...], "msg": "...", "type": "..."}] }`，但前端错误处理 `err.response?.data?.error?.message` 期望 `{ "error": { "message": "..." } }` 格式。后端自定义异常（`AppException`）有统一 handler 返回正确格式，但 Pydantic 校验错误无对应处理，导致前端只能显示通用错误消息。

**影响**: 任何 Pydantic 校验失败的请求（如类型错误、缺少字段）前端都显示模糊的通用错误，无法定位具体原因。

**修复**: 
- **后端 `exceptions.py`**: 新增 `validation_exception_handler`，将 Pydantic 的 `detail` 数组展平为 `{ "error": { "code": "VALIDATION_ERROR", "message": "loc: msg; loc: msg" } }` 格式，注册到 `EXCEPTION_HANDLERS`
- **前端 `DashboardPage.vue`**: 增强 error handler，兼容两种格式：优先读取 `error.message`，回退读取 `detail` (数组展平或字符串)

## 第五次联桥排查 — 2026-04-27

**触发**: 问题反馈 — 1. 面试官开始时不问问题，需要唤醒才问；2. 回答完问题后 AI 不回复，需要打 "?" 才有反应。

**排查结果**: 问题 13 — 前端未展示首问 + 语音输入链路三处断裂。

**分析**:

1. **首问丢失**: 后端 `interview_ws.py` 在 WebSocket 连接后依次发送 `session/started`（欢迎语）和 `control/stage_change`（含第一个问题）。`control/stage_change` 的 `data.question` 带有面试题文本，但前端 `useInterview.js` 的 `control/stage_change` handler 只使用了 `data.message`（阶段过渡文案），完全忽略了 `data.question`。用户看到的只有"接下来进入【初筛】环节"，看不到具体问题。

2. **语音链路三相断裂**:
   - **麦克风未启动**: `startInterview()` 调用了 `micInit()` 初始化音频上下文，但从未调用 `micStart()` 开启录音。`isRecording` 始终为 `false`，音频数据从未流向后端。
   - **VAD 语音结束信号空转**: 前端 `useAudioRecorder` 通过 VAD 检测到语音结束时会触发 `onSpeechEnd` 回调，但 `useInterview.js` 中此回调为空函数 `() => {}`，后端无法得知用户已说完。
   - **ASR 未接入直播循环**: 后端 `AudioHandler` 有完整的阿里云 NLS STT 实现（`speech_to_text()`），但该函数只在 `AudioHandler` 内部存在，从未被 `interview_ws.py` 的消息循环调用。音频数据通过 WebSocket 二进制帧到达后端后，仅被写入 PCM 临时文件用于事后存档，实时转写完全缺失。

3. **为什么打 "?" 有效**: 手动输入通过 `user/text` 消息类型直接进入 `orchestrator.handle_text()`，跳过了整个语音链路。

**影响**: 语音面试核心功能完全不可用。用户必须手动输入文字才能与 AI 面试官交互。

**修复**:

- **前端 `useInterview.js`**:
  - `control/stage_change` handler: 新增 `if (data.question) store.addMessage('ai', data.question)` — 将首问及每个阶段第一个问题加入对话
  - `startInterview()`: `micInit()` 成功后显式调用 `micStart()` — 自动启动录音
  - `onSpeechEnd` 回调: 从空函数改为 `send({ type: 'user/speech_end' })` — 通知后端语音结束

- **后端 `audio_handler.py`**:
  - 新增 `_utterance_buffers: dict[int, bytearray]` — 按面试 ID 隔离的语音段内存缓冲
  - `append_chunk()`: 同步写入 utterance buffer 和物理文件
  - 新增 `get_utterance_audio(interview_id)` — 取出当前语音段 PCM 数据并清空缓冲
  - `start_interview()` / `end_interview()`: 清理 utterance buffer

- **后端 `interview_ws.py`**:
  - 新增 `user/speech_end` handler: 调用 `audio_handler.get_utterance_audio()` → `audio_handler.speech_to_text()` → `orchestrator.handle_text()` → 发送响应
  - `user/interrupt` handler: 增加 `audio_handler.get_utterance_audio()` 调用以丢弃被中断的语音缓冲

**语音链路修复后流程**:
```
用户说话 → VAD 检测开始 → 发送 user/interrupt(停止TTS+清旧缓冲)
→ 音频 PCM 帧持续发送 → VAD 检测结束 → 发送 user/speech_end
→ 后端取出语音段 → 阿里云NLS STT → 文本给 orchestrator
→ 评分+决策 → AI 回复(含TTS语音) → 下一题
```

## 第八次联桥排查 — 2026-04-27

**触发**: 修复后仍然听不到 AI 说话。

**排查结果**: 问题 16 — TTS 音频格式不匹配 + 播放 AudioContext 未恢复。

**分析**:

1. **TTS 格式 WAV vs PCM**: 后端 `AliyunTTSProvider.synthesize()` 请求阿里云 NLS TTS 时使用 `format: "wav"`，返回带 44 字节 RIFF 头的 WAV 文件。前端 `pcm16ToFloat32()` 将输入直接解释为 raw PCM16 样本序列，WAV 头部被当作前 22 个样本处理（嘈杂噪声），导致音频失真。

2. **播放 AudioContext 未恢复（主因）**: 前端使用两个独立的 `AudioContext`：
   - 录音用：`useAudioRecorder.js` 中创建，通过 `getUserMedia` 的用户手势激活
   - 播放用：`useInterview.js` 中 `getAudioCtx()` 懒创建，从未调用 `resume()`
   
   浏览器要求 AudioContext 在用户手势后必须显式 `resume()`。播放用 AudioContext 创建后处于 `suspended` 状态，`source.start()` 不产生任何声音。录音 AudioContext 虽然活跃，但与播放 AudioContext 完全隔离，无法互相借用。

3. **采样率不匹配**: 播放 AudioContext 创建时使用 `sampleRate: 24000`，但 TTS 输出为 16000 Hz，导致播放速度 1.5x（音频应仍可听见，非静音主因）。

**修复**:

- **后端 `audio_handler.py`**: TTS 请求 `format` 从 `"wav"` 改为 `"pcm"`，返回原始 PCM16 数据直接供前端消费。

- **前端 `useInterview.js`**:
  - `getAudioCtx()`: 改为 `async`，创建后检查 `audioCtx.state === 'suspended'` 则调用 `await audioCtx.resume()`
  - `getAudioCtx()`: `sampleRate` 从 24000 改为 16000，匹配 TTS 输出
  - `scheduleNextChunk()`: 改为 `async`，`await getAudioCtx()` 确保 AudioContext 已恢复后再创建 buffer 播放

**修改文件**:
- `FastAPI_ai_interview/app/ws/audio_handler.py` — TTS format: wav → pcm
- `vue_ai_interview/src/composables/useInterview.js` — AudioContext resume + sample rate fix

**重要提醒**: 重启后端服务使 `audio_handler.py` 修改生效。

## 第十次联桥排查 — 2026-04-28

**触发**: 仍然是完全没有声音。

**排查结果**: 问题 18 — AudioContext API 浏览器兼容性/时序问题不可靠，整条 PCM→AudioContext 管线有太多隐式失败点。彻底重写为 `<audio>` 元素方案。

**分析**:

经过九轮修复后音频仍无输出，说明 `AudioContext` 管线（`AudioContext → createBuffer → BufferSource → connect → start`）在用户浏览器环境中存在未被捕获的静默失败。可能的失败点：

1. `AudioContext.resume()` 在无用户手势时永久悬挂（即使预先调用）
2. 浏览器不支持 `sampleRate: 16000` 的 AudioContext，静默回退到其他采样率导致 buffer/sample 不匹配
3. `AudioContext` 输出被路由到错误设备（如 HDMI/虚拟设备）
4. AudioContext 的 `destination` 在某些条件下静音
5. `source.start()` 不抛异常但不产生声音（Chrome 已知行为：AudioContext 在特定条件下静默）

根本问题：`AudioContext` API 设计用于实时音频处理（低延迟流式），而我们的场景是**播放完整的 TTS 语音片段**——这恰是 `<audio>` 元素的设计目标。

**修复（彻底重写播放管线）**:

- **新增 `audioUtils.js` — `pcm16ToWav()`**: 将原始 PCM16 数据封装为标准 RIFF/WAV 文件（44 字节头 + PCM 数据），设置正确的采样率、声道数、位深度。浏览器原生支持 WAV 解码，无需任何第三方库。

- **重写 `useInterview.js` 播放模块**（原 `AudioContext` 管线全部移除）:
  - `playNextInQueue()`: PCM16 → `pcm16ToWav()` → `Blob('audio/wav')` → `URL.createObjectURL()` → `new Audio(url).play()`
  - `queueAudioChunk()`: 仅当无活跃播放时才触发 `playNextInQueue()`
  - `flushAudioPlayback()`: 清空队列 + `activeAudio.pause()`
  - 播放完毕/出错自动释放 blob URL
  - 状态重置 `store.setAiStatus(AI_STATUS.LISTENING)` 保持在队列空时触发

**新管线的优势**:
- `<audio>` 元素使用浏览器原生解码器，支持所有音频格式
- Blob URL 方案绕过 AudioContext 的 user-activation 限制
- WAV 头明确声明格式参数，不依赖隐式假设
- 浏览器自动处理音频路由和设备选择
- 代码量从 ~45 行减至 ~35 行，无异步等待点

**修改文件**:
- `vue_ai_interview/src/utils/audioUtils.js` — 新增 `pcm16ToWav()` + `writeString()`

## 架构升级 — 2026-04-28：从题库匹配到简历驱动 + 四阶段差异化面试

**背景**: 无论选什么岗位和难度，第一题总是固定的"请描述你对LLM的理解"。追问逻辑统一，四个阶段没有实际区别。

**触发**: 1. 前端对话窗口不自动滚动到底部；2. 技术面后端仍然无限出新题，永不结束（q_in_stage 到 18/2）。

- **前端 `InterviewRoom.vue` 自动滚动**:
  - 同时使用两个 watch：一个监听 `dialogue.value?.length`（新消息），一个 deep watch 监听 `dialogue.value`（流式文本更新）

  - 新增硬限制：`if self.questions_in_stage >= max_q` → 强制 `_next_stage_message()`
  - 即使 LLM fallback 返回 `ask_question`、即使 LLM 无视 prompt 限制，代码层面锁定上限

```
用户回答 → LLM评估 → action=
```

  ```javascript
  case 'user/text_echo':

  ```

### 问题 24-A：面试官连问两题（双重 `handle_text`）

**根因分析**:

在 PTT 模式下，一次完整的语音回答产生两条处理链路：

```
用户松开录音按钮 → 前端发 user/speech_end
  → 后端 STT → user/text_echo（回显文字给前端）
  → 后端立即调用 orchestrator.handle_text()  ← 第一次处理

前端收到 user/text_echo → 创建草稿气泡
  → 4 秒后自动发送 user/text（含相同文字）
  → 后端再次调用 orchestrator.handle_text()  ← 第二次处理（重复！）
```

日志完整还原了故障时序（interview 57）：

| 时间 | 事件 |
|------|------|
| 10:22:47 | STT 返回"不知道"（3字），`handle_text("不知道")` 开始处理（初筛阶段） |
| 10:22:47 | 前端创建草稿气泡，4 秒自动发送定时器启动 |
| 10:22:51 | **前端自动发送** `user/text`（"不知道"），但此时后端仍在第一次 `handle_text` 中（等待 LLM 评估），消息被 WebSocket 缓冲 |

## 第十八次联桥排查 — 2026-04-29：STT/TTS 迁移到千问实时语音 API

**触发**: NLS 识别技术词汇几乎全挂（"unique_ptr" → "une pt r"），用户体验极差。

**方案**: 将 STT/TTS 从阿里云 NLS 迁移到百炼 DashScope 的 Qwen 实时语音 API。

### API 协议概览

两个 API 使用统一的 WebSocket 端点 `wss://dashscope.aliyuncs.com/api-ws/v1/realtime`，通过 `?model=` 区分服务，认证方式为 `Authorization: bearer <API_KEY>`。

| | Qwen-ASR-Realtime | Qwen-TTS-Realtime |
|---|---|---|
| 模型 | `qwen3-asr-flash-realtime` | `qwen3-tts-flash-realtime` |
| 协议 | OpenAI Realtime API 风格事件 | 同左 |
| 交互模式 | Manual commit（客户端控制断句） | Server commit（服务端自动分段） |
| 输入 | Base64 PCM 16kHz 16bit | 文本 |
| 输出 | 识别文字 | Base64 PCM 24kHz 16bit → 重采样到 16kHz |

### STT 交互流程（Manual 模式）

```
客户端 → session.update (turn_detection=null, pcm, 16kHz, zh)
服务端 → session.updated
```

## 第二十次联桥排查 — 2026-04-29：前端面试官无声音

**触发**: 面试过程中前端能看到面试官文字消息，但完全听不到 AI 面试官的声音。

**排查结果**: 问题 26 — `<audio>` 元素 `play()` Promise 被浏览器自动播放策略拦截，`.catch()` 静默吞掉错误。

**分析**:

完整音频链路追踪：

```
后端 TTS 生成 (DashScope Qwen-TTS) → PCM16 16kHz 字节
  → WebSocket send_bytes → 前端 onmessage ArrayBuffer
  → queueAudioChunk → pcm16ToWav → Blob('audio/wav')
  → URL.createObjectURL → new Audio(url).play()
```

链路中 TTS 生成、WebSocket 传输、WAV 封装均正常，文字消息能正常显示说明 WebSocket 连接和 LLM 对话都工作正常。问题集中在最后一环 `new Audio(url).play()`。

**根因**: `useInterview.js` 的 `playNextInQueue()` 中使用 `new Audio(url).play()` 播放 AI 语音，`.catch()` 回调为空函数，不记录任何错误信息。现代浏览器（Chrome/Edge）的自动播放策略要求 Audio 元素在用户手势上下文中创建才能自动播放。AI 语音是由 WebSocket 消息触发的异步播放，距离用户点击"开始面试"已过去数秒甚至更长，浏览器认为没有用户交互授权，`play()` 返回 rejected Promise（`NotAllowedError`），错误被静默吞掉。

**与历史修复的关系（第十次联桥排查）**:

第十次排查时曾将 `AudioContext` 管线改为 `<audio>` 元素方案，理由是 AudioContext `resume()` 在无用户手势时永久悬挂。但 `<audio>` 元素的 `play()` 同样受浏览器自动播放策略限制。两个方案都有异步播放被拦截的风险，区别仅在于 `<audio>` 的 `play()` 返回 rejected Promise 而 AudioContext 的 `resume()` 永久 pending。

**修复**:

改为 Web Audio API（AudioContext）播放方案，并在用户手势有效期内预初始化 AudioContext：

- **`useInterview.js` 播放管线重写**:
  - `pcm16ToWav` + `Blob` + `new Audio(url).play()` → **`pcm16ToFloat32` + `AudioContext.createBuffer` + `BufferSource.start()`**
  - 新增 `getAudioContext()`：懒创建 AudioContext（16kHz），自动调用 `resume()` 恢复挂起状态
  - `playNextInQueue()`: PCM16 → Float32 → AudioBuffer → connect → start，串行队列播放
  - `flushAudioPlayback()`: `activeSource.stop()` 替代 `activeAudio.pause()`，打断功能适配
  - 播放失败时 `console.error('AI音频播放失败:', e)` 输出具体错误信息，方便浏览器控制台诊断
  - `startInterview()` 中在 `micInit()` 成功后、`wsConnect()` 前预初始化 AudioContext，利用用户点击"开始面试"的手势授权
  - 播放队列为空时自动将 AI 状态设为 `LISTENING`，解除 VAD 死锁

**AudioContext 方案 vs `<audio>` 方案对比**:

| | `<audio>` 方案（旧） | AudioContext 方案（新） |
|---|---|---|
| 用户手势依赖 | 每次 `play()` 都需要手势 | 仅 `resume()` 需要，可预初始化 |
| 错误可见性 | `.catch()` 空函数，静默失败 | `console.error` 输出详细错误 |
| 资源管理 | 每次创建/销毁 DOM 元素 + Blob URL | 复用同一个 AudioContext |
| 打断控制 | `audio.pause()` | `source.stop()` |
| 采样率控制 | 依赖 WAV 头声明 | `createBuffer` 显式指定 |

**修改文件**:
- `vue_ai_interview/src/composables/useInterview.js` — 播放管线重写（AudioContext）+ 错误日志 + 预初始化

## 第二十一次联桥排查 — 2026-04-29：AudioContext resume() Promise 未 await

**触发**: 第二十次修复后前端仍无声音，状态一直卡在"正在回复"不切换。

**排查结果**: 问题 27 — `audioCtx.resume()` 返回 Promise 但未 `await`，AudioContext 保持 suspended 状态，`source.start()` 静默无声。

**分析**:

第二十次修复将播放管线从 `<audio>` 改为 `AudioContext`，但 `getAudioContext()` 中：

```javascript
// 错误写法
function getAudioContext() {
    if (!audioCtx) {
      audioCtx = new AudioContext({ sampleRate: 16000 })
    }
    if (audioCtx.state === 'suspended') {
      audioCtx.resume()  // ← 返回 Promise，但未 await！
    }
    return audioCtx
}
```

`AudioContext.resume()` 返回 `Promise<void>`，必须 `await` 等待浏览器完成音频硬件初始化。不 `await` 的情况下，后续 `source.start()` 调用时 AudioContext 仍处于 `suspended` 状态。Chrome 对 suspended AudioContext 的 `start()` 行为是**不抛异常、不输出声音、不触发 `onended`**——三重静默，无法被 catch 捕获，无法通过 `onended` 推进队列。

直接后果：
1. `source.start()` 不发声音 → 用户听不到
2. `source.onended` 永不触发 → `playNextInQueue()` 不会被再次调用
3. AI 状态永远停在 `SPEAKING`（队列未空，未触发 `setAiStatus(LISTENING)`）
4. 前端永远显示"正在回复"，用户只能手动刷新

**修复**:

- **`getAudioContext()`**: 改为 `async`，`await audioCtx.resume()`
- **`playNextInQueue()`**: 改为 `async`，`await getAudioContext()`
- **`queueAudioChunk()`**: 新增 `playScheduled` 标志防止并发调用。因为 `playNextInQueue()` 现在是 async，多次同步调用会产生竞态
- **`flushAudioPlayback()`**: 同步重置 `playScheduled = false`
- **`startInterview()`**: `micInit()` 后调用 `await getAudioContext()`，利用用户点击"开始面试"的手势授权完成 AudioContext 激活
- **错误日志增强**: 输出 `AudioContext状态: suspended/running/closed` 便于诊断

**修改文件**:
- `vue_ai_interview/src/composables/useInterview.js` — 5 处修改（getAudioContext async / playNextInQueue async / playScheduled 防竞态 / flushAudioPlayback 重置 / startInterview 预初始化）

**重要提醒**: 前端 `npm run build` 重新构建后生效。如仍无声音，打开浏览器控制台（F12）查看是否有 `AI音频播放失败:` 错误日志，根据具体错误信息进一步定位。

## 第二十二次联桥排查 — 2026-04-29：AudioContext 浏览器兼容性不可靠，改为持久 `<audio>` 元素 + 手势预解锁

**触发**: 第二十一次修复后仍无声音，状态卡在"正在回复"，控制台无任何错误信息。

**排查结果**: 问题 28 — AudioContext 在 Chrome 中的 `resume()` 行为不稳定，且 `source.start()` 在 suspended context 上静默无声不抛异常不触发 onended。

**分析**:

经过两轮 AudioContext 方案尝试后发现，Web Audio API 在不同浏览器、不同安全设置下的行为差异太大：

1. `AudioContext.resume()` 在部分 Chrome 版本中即使有用户手势也不保证成功
2. `source.start()` 在 suspended context 上：不抛异常、不发声、不触发 `onended` → 三重静默
3. 两个 AudioContext（录音+播放）可能互相干扰
4. 浏览器标签页切换到后台时 AudioContext 自动 suspended

这些问题都无法被 try-catch 捕获，`console.error` 也触发不了。前端显示"正在回复"卡死，完全无声。

**根本认知纠正**: AudioContext 是为实时音频处理（低延迟流式）设计的，不是为"播放完整语音片段"设计的。后者恰是 `<audio>` 元素的设计目标。

**彻底方案**: 

回归 `<audio>` 元素，但解决之前方案的唯一痛点——浏览器自动播放策略。核心思路：**在用户手势期间预解锁一个持久化的 `<audio>` 元素，后续所有 TTS 语音复用该元素**。

- **持久化元素**: `getAudioElement()` 懒创建一个 `new Audio()` 实例，整个面试会话复用
- **手势预解锁**: `startInterview()` 中调用 `unlockAudio()` — 播放一段 5ms 静音 WAV。此时用户刚点击"开始面试"，浏览器手势激活中，`play()` 成功。该 audio 元素被永久"信任"，后续无需用户手势即可播放
- **队列串行**: 复用原有队列机制，一次只播放一段，`onended` 驱动下一段
- **全链路诊断日志**: 每次收到音频数据、开始播放、播放成功/失败都有 `console.log/error` 输出，方便定位

```
用户点击"开始面试"
  → micInit()
  → unlockAudio() ← 在用户手势中播静音解锁 <audio> 元素
  → wsConnect()
  
WebSocket 收到 TTS 音频
  → queueAudioChunk → console.log('收到音频数据:', N, 'bytes')
  → playNextInQueue → console.log('播放队列: PCM数据大小=', N, 'bytes')
  → <audio>.src = blobUrl → .play()
  → 成功: console.log('音频播放开始成功')
  → 失败: console.error('音频play()被拒绝:', e.name, e.message)
```

**修改文件**:
- `vue_ai_interview/src/composables/useInterview.js` — 播放管线重写（持久化 `<audio>` + 手势预解锁 + 全链路诊断日志）

## 第二十三次联桥排查 — 2026-04-29：websockets 库 API 参数名不兼容

**触发**: 前端诊断日志显示"音频已解锁"但没有任何"收到音频数据"，说明 TTS 根本没生成音频。

**排查结果**: 问题 29 — `websockets.connect(extra_headers=...)` 参数名不兼容。

**分析**:

后端日志关键错误：
```
[ERROR] app.ws.audio_handler: Qwen TTS failed: BaseEventLoop.create_connection() got an unexpected keyword argument 'extra_headers'
```

`websockets` 库 13.0 版本（2024-09-07）将 `connect()` 的 `extra_headers` 参数重命名为 `additional_headers`。项目中 `requirements.txt` 指定 `websockets==16.0`，但 `audio_handler.py` 中 `QwenASRProvider.transcribe()` 和 `QwenTTSProvider.synthesize()` 仍使用旧参数名 `extra_headers`。

由于参数名不匹配，`websockets` 内部将未知关键字参数透传给了 `asyncio.BaseEventLoop.create_connection()`，后者不接受该参数，抛出 `TypeError`。异常被 catch 后返回空字节 `b""`，前端完全收不到音频数据。

TTS 和 ASR 两个 provider 都受影响（两个 `websockets.connect()` 调用均使用旧参数名）。

**修复**:

`audio_handler.py` 中两处 `websockets.connect()` 调用：
```diff
- extra_headers={"Authorization": f"bearer {self.api_key}"}
+ additional_headers={"Authorization": f"bearer {self.api_key}"}
```

涉及：`QwenASRProvider.transcribe()`（第332行）和 `QwenTTSProvider.synthesize()`（第435行）。

**修改文件**:
- `FastAPI_ai_interview/app/ws/audio_handler.py` — 2 处 `extra_headers` → `additional_headers` + `open_timeout=15, close_timeout=5`
- `FastAPI_ai_interview/app/ws/interview_ws.py` — `_send_response` 重构：文本先于 TTS 发送 + 25s 超时

## 第二十四次联桥排查 — 2026-04-29：TTS 连接 hang 死阻塞文本响应

**触发**: `extra_headers` 修复后仍无声音，且前端问题文字也不显示了（之前至少还显示），麦克风灰色，"正在回复"。

旧代码执行顺序：
```
_send_response()
  → send ai/status: speaking
  → await audio_handler.text_to_speech(text)   ← 这里如果 hang，下面全卡死
  → send control/stage_change (含题目文字)     ← 永远执行不到
```

DashScope TTS WebSocket 连接可能因为网络/防火墙/DNS 等原因 hang 在 TCP/TLS 握手阶段。`websockets.connect()` 没有显式 `open_timeout`（默认 10s），但即使超时抛出异常，异常被 catch 后返回 `b""`，文本响应依然要等超时完成后才发送。

1. **`_send_response` 顺序重构**：**先发文本响应，后生成 TTS**

   - `control/stage_change`: **send stage_change** → send speaking → TTS
   - 用户立即看到题目文字，TTS 作为后台跟进
   
2. **TTS 增加 25s 超时**：`asyncio.wait_for(text_to_speech(text), timeout=25.0)`，超时记日志但不阻塞

3. **websockets.connect 增加显式超时**：`open_timeout=15, close_timeout=5`

- `FastAPI_ai_interview/app/ws/interview_ws.py` — `_send_response` 重构

## 第二十五次联桥排查 — 2026-04-29：LLM 非 JSON 输出 + _send_response 重复消息

**触发**: DASHSCOPE_API_KEY 置空切回 NLS，前端问题文字只显示"请继续回答。"，语音混乱。

**完整链路分析**:

```
LLM (deepseek-v4-flash)  →  输出纯中文文本（非JSON）
  → llm_call_json JSON解析失败
  → fallback: {"action": "ask_question", "message": "请继续回答。", "question_text": "请继续回答。"}
  → begin_current_stage: question_text = "请继续回答。"
  → control/stage_change → 前端显示 "请继续回答。"
  → TTS 合成 "请继续回答。" → 播放 → 语音乱（内容不对）
```

**根因 1: LLM 未强制 JSON 输出**

`base.py:llm_call_json` 调用 `llm_call()` 时没有传 `response_format={"type": "json_object"}`（DeepSeek 支持 OpenAI 兼容的 JSON 模式）。DeepSeek v4-flash 在不强制 JSON 时经常在 JSON 前输出中文自然语言前言（"好的，作为面试官，我将开始初筛阶段..."），导致 JSON 解析走 fallback。Fallback 虽有 `{...}` 提取逻辑，但 LLM 偶尔完全只返回纯文本（无花括号），fallback 兜底为硬编码的 "请继续回答。"。

**根因 2: _send_response 代码重复**

上一轮修改在 `ai/text` 分支中产生了两个相同的 if/else 块（均发送 thinking），stage_change 分支也缺失了对 TTS 失败后的状态恢复。

**修复**:

1. **`base.py` — 强制 JSON + 智能 Fallback**:
   - `llm_call_json()`: 添加 `response_format={"type": "json_object"}` 强制 LLM 输出 JSON
   - 如果 messages 最后一条不含 "JSON" 关键词，自动追加 "请以JSON格式返回结果。"（`response_format` 的前置条件）
   - 新增 `_json_fallback()`: LLM 返回纯文本时剥离常见前言（"好的，作为面试官..."），用剩下的文本作为 `message` 和 `question_text`，而不是丢弃成 "请继续回答。"

2. **`interview_ws.py` — 重写 _send_response + 提取 _tts_speak**:
   - 移除重复状态消息代码
   - 提取 `_tts_speak()` 共享函数（25s 超时 + 统一错误处理）
   - 明确发送顺序：`status → text response → TTS(audio)`
   - `control/stage_change`: 先发问题文字 → 再生成 TTS
   - `ai/text`: 先发 `thinking` → 发文字 → 发 `speaking` → TTS

**修改文件**:
- `FastAPI_ai_interview/app/agents/base.py` — response_format + _json_fallback
- `FastAPI_ai_interview/app/ws/interview_ws.py` — _send_response 重构 + _tts_speak

## 第三十二次联桥排查 — 2026-05-01：面试交互五连修

**触发**: 五个独立问题集中反馈。

### 问题 32-A：用户发送消息后无即时反馈

**现象**: 用户点击发送/草稿自动发送后，没有状态变化，用户可能再次发送导致重复回答。

**根因**: `confirmDraft()` / `sendManualText()` 发送 `user/text` 后未设置 AI 状态。

**修复**: 发送前立即 `store.setAiStatus(AI_STATUS.THINKING)`。

### 问题 32-B：问题文本重复显示

**现象**: 追问和下一题有时显示两遍相同内容。例如 "很好，你对MVP的理解很准确。接下来，请问：用户故事... 用户故事..."

**根因**: LLM 返回的 `message` 字段已嵌入完整问题，`question_text` 又单独拼接一次 → 双重显示。

**修复**: `interview_orchestrator.py` 中 `follow_up` 和 `ask_question` 分支改为检测 `question_text in message`，已包含则不再拼接，只显示 `question_text`。

### 问题 32-C：沉默提示太快

**现象**: 面试官说完话，沉默提示"我还在等待你的回答..."就立即出现，没有给用户足够的思考时间。

**根因**: `_silence_prompt` 的 15 秒倒计时从文字发送时开始，包含了 TTS 播放时间（3-8 秒）→ 实际留给用户的时间只有 7-12 秒。

**修复**:
- `_silence_prompt` 增加 `is_speaking` 参数，先 `while is_speaking.is_set(): await sleep(0.5)` 等 TTS 播完，再开始 15 秒倒计时
- `_tts_player` 在队列清空后自动 `is_speaking.clear()`

### 问题 32-D：文字和语音不同步

**现象**: 前端先显示文字，隔几秒后才有语音，体验割裂。

**根因**: `_send_response` 顺序为先发文字、再生 TTS。TTS 生成需要 3-8 秒 → 文字和语音间隔过大。

**修复**: 改为先生成 TTS → 再发文字 + 队列音频。TTS 失败时发送空状态，不影响文字显示。`_tts_speak` 返回值改为 `bool`。

## 第四十次联桥排查 — 2026-05-03：音频回放 401 Token 验证失败

### 问题 29：Sub 字段类型不兼容

**现象**: 录音文件保存成功（`Audio merged and uploaded`），但前端请求音频 URL 时返回 401 `Invalid or expired token`。

**根因**: `create_file_token()` 中 `sub` 字段传入了 `int` 类型的 `user_id`。旧库 `python-jose` 对此宽松，但迁移到 PyJWT 2.12 后，严格按照 RFC 7519 要求 `sub`（Subject）必须是字符串。日志：

```
Token created, length: 223
Decode FAILED: Subject must be a string
```

**修复**:
```diff
# security.py: create_file_token()
- {"sub": user_id, "file": filename, "type": "file_access", "jti": ..., "exp": expire}
+ {"sub": str(user_id), "file": filename, "type": "file_access", "jti": ..., "exp": expire}
```

**三层 bug 链回顾**（同一个录音回放功能）：

| 层 | 错误 | 修复 |
|----|------|------|
| 1 | `merge_and_upload` 未接收 `user_id` → NameError | 增加参数 + `end_interview` 传递 |
| 2 | Docker 绑定挂载 `./uploads` 权限为 root → 写入失败 | 改为命名卷 `uploads_data` |
| 3 | `create_file_token` 的 `sub` 为 int → PyJWT 拒绝 | 改为 `str(user_id)` |

**修改文件**:
- `FastAPI_ai_interview/app/core/security.py` — `user_id` → `str(user_id)`

## 第四十一次联桥排查 — 2026-05-04：iPhone 无法访问 + 邀请码显示异常

### 问题 30-A：iPhone Safari 打不开站点

**现象**: 其他设备正常，唯独 iPhone 上 Safari 和 Chrome 都打不开 `https://www.aiview.pandahead.top`，无报错白屏。

**根因**: `index.html` 的 CSP meta 标签中 `connect-src` 只配了 `'self'` + `localhost:*` + `wss://*:*`。iPhone Safari 对 CSP 解析更严格，且生产环境不需要 `localhost`。缺少明确的 `https:` 和 `wss:` 协议授权，导致所有 API/WS 请求被拦截。

**修复**: CSP `connect-src` 增加 `https:` 和 `wss:` 协议前缀：
```diff
- connect-src 'self' http://localhost:* ws://localhost:* wss://*:*
+ connect-src 'self' https: wss: http://localhost:* ws://localhost:*
```

### 问题 30-B：邀请码显示 ---

**现象**: 管理员登录后打开邀请码弹窗，显示 `---`。重新登录一次后才正常显示 `DAA835F512C8`。

**根因**: `AppLayout.vue` 中邀请码 API 调用失败（Token 过期或网络问题）时，catch 块直接写死 `inviteCode.value = '---'`，不区分是 401 还是其他错误，用户不知道是"需要重登"还是"服务器故障"。

**修复**: catch 块按错误类型区分提示：
```diff
- } catch {
-   inviteCode.value = '---'
+ } catch (err) {
+   inviteCode.value = err.response?.status === 401 ? '请重新登录' : '获取失败'
```

**修改文件**:
- `vue_ai_interview/index.html` — CSP 增加 https: wss:
- `vue_ai_interview/src/components/AppLayout.vue` — 邀请码错误提示优化

**重要提醒**: 前端 `npm run build` + `sudo cp -r dist/* /var/www/ai_interview/` 生效。
